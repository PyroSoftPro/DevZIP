Windows-first archival engine

DevZip

DevZip is an experimental .dvz container focused on size-first compression for mixed large datasets. The goal is to beat strong desktop archivers on representative corpora while keeping the shipping engine commercially usable. Product success is measured by end size first; Weissman stays as a secondary comparison only.

Current status

Format v3 ships LZMA2-compressed manifests with a best-of-two block-compression backend (LZMA2 vs ZPAQ M4, winner per solid group), solid-packing by file type, and parallel compression across both codecs and all solid groups. Experimental best-of-three-ppmd and selective-zpaq5 lanes are now available via devzip_cli.exe --backend ...; all still emit readable .dvz archives.

devzip-native beats 7z-lzma2 by ~1 MB on the aggregate corpus (143.815 MB vs 144.841 MB, -0.71%) in 304.2s. The latest candidate bake-off says best-of-three-ppmd is effectively a tie on large-text and a small win on large-code (-0.22%), while selective-zpaq5 is the best measured code-heavy variant so far (150.442 MB, -0.73% vs best-of-two). Aggregate mixed-large has not yet been rerun with those candidates, so the shipping default stays best-of-two.

What changed

best-of-three-ppmd adds a third PPMd codec contender to the current LZMA2 vs ZPAQ M4 bake-off. selective-zpaq5 keeps the shipping routing, but raises the ZPAQ side from M4 to M5 for text- and raw-like solid groups only.

What it means

PPMd does not improve the large text corpus at all in this pipeline, but it does help code-heavy data a little. The stronger ZPAQ routing wins the current code-heavy bake-off by 1,159,746 bytes, so it is the next candidate worth rerunning on mixed-large before any default is changed.

Key findings

devzip-native wins on every category vs 7z-lzma2: text (-32.7%), raw bitmaps (-23.8%), JPEG (-1.5%), PNG (-1.0%), video (-0.9%), random (-0.01%). Software binaries are near-tied (+0.32%).

The shipping best-of-two backend still removes the old regressions and stays the default until the code-heavy wins generalize. In the latest native bake-off, best-of-three-ppmd is a text tie and a small code improvement, while selective-zpaq5 is the strongest measured code-heavy lane (150.442 MB, -0.73% vs best-of-two on large-code).

Development notes

What currently ships

  • A native C++ engine and CLI with a best-of-two backend that competes LZMA2 and ZPAQ M4 per solid group, auto-selecting the smaller result; experimental best-of-three-ppmd and selective-zpaq5 lanes are available via --backend, and all of them write readable .dvz archives.
  • A WPF launcher that shells out to devzip_cli.exe.
  • A corpus benchmark harness for tracking aggregate end size, timing, and optional Weissman context.

Quick start

Run the benchmark harness

Build devzip_cli.exe once before running the full matrix so the devzip-native lane is included in the results.

Build the native lane

cmake --build native/engine/build --target devzip_cli devzip_tests

Basic run

python benchmarks/run_benchmarks.py --manifest benchmarks/manifests/mixed-large.json

Write the markdown report too

python benchmarks/run_benchmarks.py --manifest benchmarks/manifests/mixed-large.json --markdown-out docs/benchmarks/baseline-results.md

Override tool paths

python benchmarks/run_benchmarks.py --manifest benchmarks/manifests/mixed-large.json --tool 7z-lzma2="C:\Program Files\7-Zip\7z.exe" --tool winrar="C:\Program Files\WinRAR\Rar.exe"

Tooling boundary

What stays out of shipping binaries

7-Zip algorithm lanes, WinRAR, and native Windows ZIP are benchmark peers only. They must not be linked into shipping binaries.

The native engine defaults to the best-of-two backend, which compresses each solid group with both LZMA2 and ZPAQ M4 and keeps the smaller result. Experimental native lanes such as best-of-three-ppmd and selective-zpaq5 stay inside the .dvz engine and remain benchmark candidates until a full mixed-corpus rerun clears them.

Weissman scoring remains enabled in the harness, but the benchmark gate is still based on aggregate end size only.
