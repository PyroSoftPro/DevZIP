Windows-first archival engine

DevZip

DevZip is an experimental .dvz container focused on size-first compression for mixed large datasets. The goal is to beat strong desktop archivers on representative corpora while keeping the shipping engine commercially usable. Product success is measured by end size first; Weissman stays as a secondary comparison only.

Current status

Format .dvz v4 ships per-format recompressors (brunsli JPEG transcode, preflate deflate-undo), architecture-aware BCJ filters, similarity-ordered solid packing, and a best-of-N backend that competes LZMA2, ZPAQ-5, PPMd, and libbsc BWT per solid group, keeping the smallest.

The overhaul flipped the categories that used to lag: JPEG (-1.5% -> -21.9%), PNG (-0.95% -> -16.4%), and binaries (+0.32% loss -> -4.6% win), moving the aggregate from break-even into a double-digit win. All transforms self-verify at compress time and round-trip byte-exact.

Levels

Speed / size tradeoff

--level selects how hard the engine works. The container format is identical across levels; higher levels enable more (and more expensive) recompressors and backends. Default is balanced.

Level Transforms Backend pool Use when
fastBCJ + delta filtersLZMA2Speed first
balanced+ brunsli, code dict, PNG IDAT stripLZMA2 + ZPAQ-5Wins every category, no 5-minute PNG path
max+ preflate (PNG / zip / gzip)+ PPMdSmallest size
insane+ create-time roundtrip verify+ libbsc BWTSmallest size, paranoid verification

Compression time (seconds)

DevZIP trades time for size: ZPAQ-5 context mixing and the preflate PNG path are the dominant costs. Codecs run concurrently per solid group and groups compress in parallel across cores, so wall-time tracks the slowest codec rather than their sum.

What changed in the overhaul

New per-format recompressors (brunsli JPEG, preflate deflate-undo), architecture-aware BCJ for PE binaries, a content-adaptive LZMA2 literal model, similarity-ordered solid packing, and a best-of-N backend (LZMA2 / ZPAQ-5 / PPMd / libbsc).

What it means

DevZIP now leads 7-Zip in every measured category and by -18.0% in aggregate at max. Use --level fast when speed matters more than the last few percent of size.

Why DevZIP wins

Content-aware transforms, not just a stronger coder: brunsli beats JPEG XL (cjxl) on JPEG (19.018 vs 20.912 MB) and preflate beats zopflipng on PNG (17.550 vs 19.629 MB) while staying byte-exact.

The honest gap

On raw code/exe streams, kanzi -l9 (Apache-2.0 context mixing) edges DevZIP's current backend, and the paq8px ceiling on code is 0.109 MB. A kanzi-class CM codec is the next best-of-N candidate. See docs/benchmarks/competitive-landscape.md.

Key findings

On the overhaul sweep, DevZIP now leads 7z-lzma2 in every category: text (-40.7%), JPEG (-21.9%), PNG (-16.4%), code (-16.1%), executables (-4.6%) — aggregate -18.0%.

The pre-overhaul reference categories above (raw bitmaps, video, random) were not re-measured; raw bitmaps remained the original strong win (~-24%) via the delta filter, and high-entropy random stays at parity as expected.

Development notes

What currently ships

  • A native C++ engine and CLI with a best-of-N backend that competes LZMA2, ZPAQ-5, PPMd, and libbsc BWT per solid group, auto-selecting the smallest; per-format recompressors (brunsli JPEG, preflate deflate-undo) and architecture-aware BCJ filters all write readable .dvz v4 archives.
  • --level fast | balanced | max | insane tunes the speed/size tradeoff (default balanced), exposed in both the CLI and the WPF UI.
  • A WPF launcher that shells out to devzip_cli.exe.
  • A corpus benchmark harness for tracking aggregate end size, timing, and optional Weissman context.

Quick start

Run the benchmark harness

Build devzip_cli.exe once before running the full matrix so the devzip-native lane is included in the results.

Build the native lane

cmake --build native/engine/build --target devzip_cli devzip_tests

Basic run

python benchmarks/run_benchmarks.py --manifest benchmarks/manifests/mixed-large.json

Write the markdown report too

python benchmarks/run_benchmarks.py --manifest benchmarks/manifests/mixed-large.json --markdown-out docs/benchmarks/baseline-results.md

Override tool paths

python benchmarks/run_benchmarks.py --manifest benchmarks/manifests/mixed-large.json --tool 7z-lzma2="C:\Program Files\7-Zip\7z.exe" --tool winrar="C:\Program Files\WinRAR\Rar.exe"

Tooling boundary

What stays out of shipping binaries

7-Zip algorithm lanes, WinRAR, and native Windows ZIP are benchmark peers only. They must not be linked into shipping binaries.

The native engine's balanced default competes LZMA2 and ZPAQ-5 per solid group and keeps the smaller result; max and insane add PPMd and libbsc BWT to the pool. All codecs and recompressors stay inside the .dvz engine and are permissively licensed.

Weissman scoring remains enabled in the harness, but the benchmark gate is still based on aggregate end size only.
